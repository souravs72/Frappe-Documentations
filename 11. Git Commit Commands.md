# Git Commit Types Guide

This guide provides a concise and clear reference for writing **meaningful Git commit messages**. It includes commit types, their purpose, and short examples. Use this as a reference to make your commits consistent, readable, and easy to understand.

---

## 1. Feature (feat)

**Purpose:** Adding a new feature or functionality.

**Format:**

```
feat: <short description>
```

**Examples:**

* `feat: add recurring task generation for CG Task Definition`
* `feat: implement subtask creation in task instance`

---

## 2. Bug Fix (fix)

**Purpose:** Fixing a bug or error in the code.

**Format:**

```
fix: <short description>
```

**Examples:**

* `fix: resolve missing recurrence_type_id column error`
* `fix: correct PDF download for mis score`

---

## 3. Refactor (refactor)

**Purpose:** Code changes that **do not affect behavior** but improve structure, readability, or maintainability.

**Format:**

```
refactor: <what was refactored>
```

**Examples:**

* `refactor: simplify generate_recurring_task_instances logic`
* `refactor: extract helper functions for subtask creation`
* `refactor: clean up imports and formatting in cg_task_definition.py`

---

## 4. Documentation (docs)

**Purpose:** Adding or updating documentation.

**Format:**

```
docs: <what was documented>
```

**Examples:**

* `docs: add instructions for recurring task cron schedule`
* `docs: update README for task instance generation`

---

## 5. Style (style)

**Purpose:** Changes that **do not affect functionality** (formatting, spacing, naming conventions).

**Format:**

```
style: <what was changed>
```

**Examples:**

* `style: fix indentation and line breaks in cg_task_definition.py`
* `style: remove unused imports in clapgrow_app`

---

## 6. Test (test)

**Purpose:** Adding or updating tests.

**Format:**

```
test: <what was tested>
```

**Examples:**

* `test: add unit tests for generate_recurring_task_instances`
* `test: create test cases for subtask instance creation`

---

## 7. Chore (chore)

**Purpose:** Maintenance tasks that do not modify src or tests.

**Format:**

```
chore: <what was done>
```

**Examples:**

* `chore: update dependencies for clapgrow_app`
* `chore: migrate database to add recurrence_type_id`

---

## 8. Performance (perf)

**Purpose:** Improving performance or efficiency.

**Format:**

```
perf: <what was optimized>
```

**Examples:**

* `perf: optimize database query in recurring task generation`
* `perf: cache tag data for faster retrieval`

---

## 9. Build / CI / Deployment (build / ci / deploy)

**Purpose:** Changes related to build scripts, CI configuration, or deployment.

**Format:**

```
build: <what was changed>
ci: <what was changed>
deploy: <what was changed>
```

**Examples:**

* `build: update bench scripts for recurring task cron`
* `ci: fix GitHub Actions workflow for clapgrow_app`
* `deploy: adjust server cron for one-time execution`

---

### âœ… Tips for Writing Good Git Commit Messages

1. Use **imperative mood**: "Add feature" instead of "Added feature".
2. Keep the **summary short** (50-72 characters).
3. Explain **why**, not just what, in the optional body.
4. Use a **prefix (type)** for clarity: `feat`, `fix`, `refactor`, etc.
5. Keep commit messages **consistent** across the team.

---

**Example Workflow:**

```
git add .
git commit -m "refactor: clean up generate_recurring_task_instances function"
git push origin main
```

This guide ensures that your Git history is **clean, readable, and easy to navigate**.
